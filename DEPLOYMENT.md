# Deploying PRIMO to Render

Two web services from one blueprint: `primo-api` and `primo-web`.
Everything below is already encoded in [`render.yaml`](./render.yaml) — this
file explains the decisions and the parts you have to do by hand.

---

## Before you touch Render

**Rotate the MongoDB Atlas password.** The old one is still readable in this
repository's git history, and it is a public repo. You are about to paste a
connection string into a dashboard, so rotate it first and paste the new one —
otherwise you have simply copied a known-compromised credential somewhere else.

---

## 1. Region — the decision that matters most

The Atlas cluster resolves to **Frankfurt**:

```
ac-t74vlex-shard-00-00.aitccfh.mongodb.net → 65.62.83.98 → Frankfurt, DE
```

Both services are pinned to `region: frankfurt` to sit beside it. Render
defaults to **Oregon**; leaving that default puts every database query on a
round trip across the Atlantic, and a single page render makes several. That
alone is the difference between a fast store and an unusable one.

From Saudi Arabia expect ~100–130 ms. No major host offers a Gulf region, so
Frankfurt is the best available option either way.

## 2. Plan — do not use the free tier

Both services are on `starter`. A free service sleeps after ~15 minutes idle
and the next visitor waits ~50 s while it wakes. For a link handed to a client,
that reads as "the site is broken".

Roughly $7/service/month.

## 3. Deploy

1. Render Dashboard → **New → Blueprint** → pick this repo. It reads
   `render.yaml` and proposes both services.
2. Fill in the values marked `sync: false` (see the table below).
3. Apply. First build takes a few minutes.
4. Atlas → **Network Access** → allow Render's outbound IPs (or `0.0.0.0/0`
   for a demo). Without this the API boots and then fails every query.

### Values you must supply

| Service | Variable | Value |
|---|---|---|
| api | `MONGODB_URI` | The **rotated** Atlas connection string |
| api | `CORS_ORIGIN` | `https://primo-web.onrender.com` — full origin, **no** trailing slash |
| web | `NEXT_PUBLIC_API_URL` | `https://primo-api.onrender.com/api/v1` — the `/api/v1` suffix is required |
| web | `NEXT_PUBLIC_APP_URL` | `https://primo-web.onrender.com` |

`JWT_ACCESS_SECRET` and `JWT_REFRESH_SECRET` are generated by Render. They must
be ≥32 characters and different from each other; the API refuses to boot in
production otherwise, by design.

### Chicken-and-egg on the URLs

Neither service knows the other's URL until both exist. Deploy once, copy the
two `.onrender.com` hostnames, set the four variables above, then redeploy.

`NEXT_PUBLIC_*` variables are compiled into the browser bundle at **build**
time, so changing `NEXT_PUBLIC_API_URL` needs a full rebuild — a restart will
silently keep the old value.

## 4. Payments in this deployment are SIMULATED

The blueprint sets:

```
PAYMENT_PROVIDER=demo
ALLOW_DEMO_PAYMENTS=i-understand-no-money-is-collected
```

Card and Apple Pay orders are marked **paid without any money being
collected**. This exists so the checkout journey can be demonstrated before a
payment provider is contracted.

`NODE_ENV=production` normally refuses the demo provider outright. The
acknowledgement string is the deliberate second step that overrides it — it is
an exact sentence rather than `=true` precisely so it cannot be switched on by
accident or by a generic "turn everything on" sweep.

`GET /api/v1/payments/availability` returns `simulated: true` while this is
active, so the storefront can label it.

**Before accepting a single real order:**

1. Delete `ALLOW_DEMO_PAYMENTS`
2. Set `PAYMENT_PROVIDER=moyasar`
3. Add `MOYASAR_SECRET_KEY` and `MOYASAR_WEBHOOK_SECRET`
4. Point the Moyasar webhook at `https://<api-host>/api/v1/payments/webhook`

Leaving it as-is on a live store means giving the products away.

## 5. Things that are easy to miss

**The demo shares the production database.** The deployed site talks to the
same Atlas cluster as local development, so anything a client does — orders,
reviews, edits — shows up locally too, and vice versa. Fine for a demo; give it
its own cluster before it is a real store.

**Email is optional.** With `SMTP_*` unset, order confirmations simply are not
sent. Nothing breaks. (The credentials currently in `.env` are rejected by the
provider with `535 auth failed`, so email does not work locally either.)

**VAT invoices are not yet legally valid.** The seller VAT number is unset in
Settings, so the ZATCA QR is structurally correct but incomplete. Fill it in
under Admin → Settings before the invoices are used for anything real.

**Four admin toggles do nothing.** `stripeEnabled`, `paypalEnabled`,
`enablePushNotifications` and `enableMarketingEmails` exist in the settings UI
with no implementation behind them. They are left off deliberately — switching
them on changes nothing and would mislead anyone demoing the panel.

**Maintenance mode is safe to use.** It returns 503 for storefront writes while
leaving browsing, `/auth`, `/settings` and the whole admin panel working, and
staff bypass it — so enabling it cannot lock you out of the switch that turns
it off.
